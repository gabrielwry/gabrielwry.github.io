# DDIA Reading Notes (Part 3)

This is the third part of my reading notes on "Designing Data-Intensive Applications" by Martin Kleppmann.

## Replication

Replication means keeping a copy of the same data on multiple machines connected via a network. There are several reasons why you might want to replicate data:

- **High availability**: Keep the system running even if one machine fails
- **Disconnected operation**: Allow the application to continue working when there's a network interruption
- **Latency**: Place data geographically close to users
- **Scalability**: Handle more read requests than a single machine can handle

### Leaders and Followers

The most common solution for replication is called *leader-based replication* (also known as master-slave replication):

1. One replica is designated as the **leader** (master/primary)
2. Other replicas are **followers** (read replicas, slaves, secondaries)
3. Writes go to the leader first
4. The leader sends the data change to all followers as part of a replication log

## Consistency Models

Different consistency models provide different guarantees:

### Strong Consistency
After a write completes, all subsequent reads will see that write. This is the easiest to reason about but has performance implications.

### Eventual Consistency
If no new updates are made, eventually all replicas will converge to the same value. This provides better performance but can be confusing.

### Read-your-writes Consistency
Users should always see their own updates immediately, but other users' updates may be delayed.

## Key Takeaways

- Replication is essential for reliability and performance
- Different consistency models offer different tradeoffs
- Understanding these tradeoffs is crucial for system design
